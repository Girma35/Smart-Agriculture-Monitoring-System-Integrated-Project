SLIDE 4 — SYSTEM ARCHITECTURE OVERVIEW
================================================================
Title: How the System is Built (Big Picture)

The system has three main parts that work together:

  1. HARDWARE (The Eyes on the Ground)
     - ESP32 microcontroller embedded in the farm
     - Sensors that measure soil moisture, temperature, humidity
     - A physical water pump connected to the system

  2. BACKEND (The Brain — My Role)
     - A cloud server that receives sensor data from the farm
     - Processes the data, stores history, and runs AI analysis
     - Sends commands back to the pump (ON or OFF)
     - Deployed live on the internet at Render.com

  3. MOBILE/WEB APP (The Face)
     - Dashboard that the farmer sees on their phone or computer
     - Displays live data, diagnosis results, and alerts

My role was building the Backend (the Brain) and the AI Services.

SLIDE 5 — THE BACKEND EXPLAINED SIMPLY
================================================================
Title: What is the Backend? (In Simple Terms)

Imagine a restaurant:
  - The customer (Mobile App) places an order
  - The waiter (Backend) takes that order to the kitchen
  - The kitchen (Database / AI) prepares the response
  - The waiter delivers it back to the customer

The Backend is the waiter.
It receives requests, processes them, and sends back answers.

In our system, the Backend:
  - Listens for sensor data arriving from the farm every few seconds
  - Answers questions from the mobile app (e.g., "What is the latest reading?")
  - Runs AI analysis on the data and sends back recommendations
  - Accepts images from the app and runs disease detection on them

Technology Used: Node.js + Express.js
Deployed On: Render.com (live on the internet)
Live URL: https://smart-agriculture-backend-2nk0.onrender.com

SLIDE 7 — FEATURE 1: LIVE SENSOR DATA MONITORING
================================================================
Title: Real-Time Farm Monitoring

What it does:
  The ESP32 device in the field continuously measures:
    - Soil Moisture (%) — Is the soil wet or dry?
    - Temperature (°C)  — Is it too hot or cold for crops?
    - Humidity (%)      — Is the air too dry or too moist?
    - Pump Status       — Is the water pump currently ON or OFF?

How the data travels:
  ESP32 Sensor → Sends data via MQTT protocol → Backend receives it
  → Mobile App displays it → Farmer sees it in real-time

MQTT (Message Queuing Telemetry Transport) is a lightweight
communication protocol designed for IoT devices. It's the same
technology used in smart home devices and industrial machines.

Our MQTT Broker: HiveMQ (a free, reliable cloud MQTT server)
Topic for data: astu/agri/data
Topic for pump: astu/agri/control

The dashboard refreshes every 5 seconds automatically.

SLIDE 8 — FEATURE 2: AUTOMATED PUMP CONTROL
================================================================
Title: Smart Water Management — The Pump Control System

The Problem Without Automation:
  Farmers manually check soil moisture and decide when to irrigate.
  This takes time, wastes water, and can result in crop damage.

Our Solution:
  The backend automatically monitors soil moisture every few seconds.
  When the soil becomes too dry, the system sends an "ON" command
  to the water pump. When moisture is restored, it sends "OFF".

How it works technically:
  1. Sensor sends soil moisture reading (e.g., 22%)
  2. Backend compares it to the safe threshold (30%)
  3. Since 22% < 30%, backend publishes "ON" to the MQTT control topic
  4. The ESP32 receives the command and activates the pump relay
  5. Water flows. Soil moisture increases.
  6. When moisture rises above 60%, the backend sends "OFF"

Manual Override:
  The farmer can also manually control the pump from the app.
  They press "Pump ON" or "Pump OFF" in the dashboard.

Water Savings:
  By only pumping when necessary, the system significantly
  reduces water usage compared to scheduled or manual irrigation.

SLIDE 9 — FEATURE 3: AI-POWERED SENSOR ANALYSIS
================================================================
Title: Intelligent Farming Recommendations

Beyond just displaying numbers, the backend analyzes the sensor
readings and generates specific farming advice.

Example: If soil moisture is 22%, temperature is 38°C:
  → "Soil moisture (22%) is critically low. Irrigation required immediately."
  → "Temperature exceeds safe crop limit. Increase irrigation slightly
     to assist plant transpiration cooling."
  → "Hold fertilizer application. Stabilize soil moisture first."

How it works:
  We built a rules engine inside the backend. This engine knows
  the ideal growing ranges for 6 Ethiopian crops:

    Crop     | Ideal Temp | Ideal Humidity | Ideal Soil Moisture
    ---------|------------|----------------|---------------------
    Maize    | 18–32°C    | 50–80%         | 40–70%
    Tomato   | 18–29°C    | 60–80%         | 50–75%
    Teff     | 10–27°C    | 40–70%         | 30–60%
    Wheat    | 12–25°C    | 40–70%         | 40–65%
    Sorghum  | 20–35°C    | 40–75%         | 30–65%
    Barley   | 10–25°C    | 45–70%         | 35–60%

  When readings fall outside these ranges, the system classifies
  the situation as "optimal," "warning," or "critical" and
  provides matching advice.

This feature runs entirely for FREE — no paid AI API needed.

SLIDE 11 — HOW THE AI WAS TRAINED (EXPLAINED SIMPLY)
================================================================
Title: Teaching the Computer to Recognize Plant Diseases

Training an AI model is like teaching a child to recognize animals.
You show them hundreds of pictures and tell them "this is a cat,
this is a dog." Eventually, they learn to identify on their own.

We did the same thing — but for plant diseases.

Step 1 — Choosing the Platform:
  We used Roboflow, a professional AI platform that specializes
  in teaching computers to "see" and interpret images.
  It is trusted by researchers, universities, and companies worldwide.

Step 2 — Building the Training Dataset:
  The AI was trained on thousands of crop leaf images, including:
    - Healthy leaves (so it knows what "normal" looks like)
    - Leaves infected with Common Rust
    - Leaves infected with Blight
    - Leaves infected with Leaf Spot
    - Leaves infected with Powdery Mildew
    - Leaves infected with Mosaic Virus

Step 3 — Annotation (Teaching Labels):
  Human experts manually labeled each image, drawing boxes around
  the diseased spots and tagging them with the correct disease name.
  This is called "annotation" — it is how the AI learns.

Step 4 — Training the Model:
  Roboflow fed all labeled images into a deep learning algorithm.
  After processing thousands of examples, the AI learned to identify
  the unique colors, shapes, and textures of each disease.

Step 5 — Deploying the Model:
  Once trained, the model was published to Roboflow's cloud.
  Our backend calls this model whenever a farmer uploads an image.

SLIDE 12 — HOW WE CUSTOMIZED THE AI FOR OUR SYSTEM
================================================================
Title: Making the AI Useful, Not Just Smart

The AI alone only tells you "what disease it is."
That is not enough for a farmer. They also need to know
"what do I do about it?"

Our Customization:
  We built a local disease knowledge database directly into
  the backend. This database maps each disease name to:

    - A specific remedy (what chemicals/actions to apply)
    - A prevention tip (how to avoid it in the future)
    - A description (what the farmer should know)

Our Disease Database (Built Into the Backend):

  1. RUST DISEASE
     Remedy: Apply copper-based fungicides immediately.
             Remove and destroy severely affected leaves.
     Prevention: Plant rust-resistant hybrids, rotate crops,
                 avoid overhead sprinkler watering.

  2. LEAF SPOT DISEASE
     Remedy: Apply triazole or chlorothalonil fungicides.
             Clear all fallen debris from the field.
     Prevention: Ensure plant spacing for airflow, rotate crops annually.

  3. BLIGHT DISEASE
     Remedy: Apply copper-based or systemic fungicides early.
             Prune lower leaves to reduce soil splash infection.
     Prevention: Rotate crops, use certified disease-free seeds,
                 avoid watering late in the day.

  4. POWDERY MILDEW
     Remedy: Apply sulfur-based fungicides or organic neem oil.
     Prevention: Grow in full sun exposure, prune for aeration.

  5. MOSAIC VIRUS
     Remedy: No chemical cure. Remove and burn infected plants.
     Prevention: Control aphid insects, plant certified stock.

  6. HEALTHY PLANT
     Remedy: No treatment needed.
     Prevention: Continue regular care and monitoring.

This customization turns a raw AI result into actionable farm advice.

SLIDE 13 — THE DIAGNOSIS FLOW: END TO END
================================================================
Title: What Happens When You Scan a Leaf (Full Journey)

Step 1 — Farmer Opens the App
  The farmer notices yellowing or spots on a leaf.
  They open the Smart Agriculture app on their phone.

Step 2 — Upload the Image
  The farmer selects the "AI Leaf Diagnosis" section.
  They tap "Upload Image" and take or select a leaf photo.

Step 3 — Image Travels to the Backend
  The photo is compressed and sent securely to our backend server
  running at: https://smart-agriculture-backend-2nk0.onrender.com

Step 4 — Backend Validates the Image
  The backend checks:
    ✅ Is a file actually attached?
    ✅ Is it an accepted format? (JPG, PNG, WEBP, GIF only)
    ✅ Is the file not too large? (max 10MB)
  If anything fails, the farmer gets a clear error message.

Step 5 — Image Sent to the AI (Roboflow)
  The backend converts the image to base64 format and sends it
  to the Roboflow inference API in the cloud.

Step 6 — AI Returns Predictions
  Roboflow analyzes the leaf and returns a list of "predictions"
  — each prediction has a disease name and a confidence score.

Step 7 — Backend Picks the Best Result
  Our backend selects the prediction with the highest confidence.
  It then looks up the disease in our local knowledge database
  and attaches the matching remedy and prevention advice.

Step 8 — Severity is Calculated
  The backend determines severity based on confidence:
    - Confidence above 85% → HIGH severity
    - Confidence above 55% → MEDIUM severity
    - Below 55%            → LOW severity

Step 9 — Full Result Sent Back to the App
  The farmer sees:
    - Disease name, confidence score, severity level
    - Remedy (what to do now)
    - Prevention (how to avoid it in future)

Total time: Under 5 seconds.

SLIDE 14 — SENSOR HISTORY LOG FEATURE
================================================================
Title: Tracking Farm Trends Over Time

What it does:
  The backend stores a rolling log of the last 50 sensor readings.
  This history is always available to the mobile app.

What the history shows:
  For each reading in the history log:
    - Exact timestamp (when it was recorded)
    - Soil Moisture percentage
    - Temperature in Celsius
    - Humidity percentage
    - Pump Status (was the pump ON or OFF at that time?)

Color Coding in the Dashboard:
    - RED readings   → Soil moisture below 30% (CRITICAL — dry!)
    - ORANGE readings → Soil moisture 30–60% (OK — monitor closely)
    - GREEN readings  → Soil moisture above 60% (GOOD)

Why this matters for farmers:
  By looking at history, farmers can answer questions like:
    - "How quickly does my soil dry out on hot days?"
    - "What time of day is the pump activating most often?"
    - "Has the temperature been consistently too high this week?"

This data-driven insight helps farmers plan irrigation schedules
and make better long-term decisions for their farm.

SLIDE 16 — DEPLOYMENT: FROM LOCAL TO LIVE
================================================================
Title: How We Deployed the Backend to the Internet

Originally, the backend ran only on a local computer (localhost).
This means only someone in the same room could access it.

For the real product, the backend needs to be accessible from
anywhere in the world — from the farmer's phone, from the web app,
and from the IoT sensor in the field.

Deployment Steps:
  1. We created a free account on Render.com
  2. We connected our project's GitHub repository to Render
  3. We configured the start command: "node index.js"
  4. We set the environment variables (secret keys like API keys)
  5. Render automatically built and launched the server
  6. Within minutes, the backend was live at our public URL

Environment Variables (Secret Configuration):
  These are private keys stored securely on the server:
    - ROBOFLOW_API_KEY  → The secret key to access the AI
    - ROBOFLOW_MODEL_ID → Which AI model to use for diagnosis
    - PORT              → The port the server runs on

The Render URL (Live & Working):
  https://smart-agriculture-backend-2nk0.onrender.com

This deployment means the entire system works even if our
computers are turned off — the server runs independently in the cloud.

SLIDE 17 — SECURITY & RELIABILITY
================================================================
Title: How We Made the System Safe and Reliable

Security Measures:

  1. Input Validation
     Every request is checked before being processed.
     Example: If someone sends a non-image file to the diagnosis
     endpoint, the backend rejects it with a clear error message.
     Only JPG, PNG, WEBP, and GIF files are accepted.

  2. Environment Variables
     Secret API keys are never written directly in the code.
     They are stored as environment variables on the server,
     making it impossible for them to be accidentally exposed.

  3. Error Handling
     Every function in the backend is wrapped in a try/catch block.
     If anything goes wrong, the system returns a clean error message
     instead of crashing. The farmer always gets a useful response.

  4. CORS Protection
     CORS (Cross-Origin Resource Sharing) settings control which
     apps are allowed to talk to the backend. This prevents
     unauthorized websites from accessing our server.

Reliability Features:

  1. Automatic Reconnection
     If the MQTT connection drops, the backend automatically
     tries to reconnect every 5 seconds. No manual restart needed.

  2. Fallback Data
     If no live sensor reading has arrived yet, the backend
     returns a realistic placeholder reading so the app doesn't
     crash or show empty data.

  3. Rolling History Buffer
     History is stored in memory with a maximum of 50 entries.
     Older entries are automatically removed to prevent memory overload.
